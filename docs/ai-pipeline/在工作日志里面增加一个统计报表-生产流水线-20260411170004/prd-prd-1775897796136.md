## 1. 文档概述

### 1.1 文档背景
当前工作日报系统缺乏有效的日志填写情况监控与分析工具。管理者（如部门负责人、HR、高管）无法便捷、量化地掌握团队或员工的日志填写情况，存在以下核心痛点：
1.  **管理效率低下**：管理者需人工逐一检查员工日志或通过非正式渠道询问，耗时耗力，难以实现规模化、周期性的管理。
2.  **数据感知滞后**：无法及时发现员工填写率低、拖延提交等问题，导致管理动作滞后，影响工作复盘与过程管理效果。
3.  **缺乏决策依据**：缺少客观数据支持，难以评估团队协作效率、员工工作饱和度，也无法为绩效考核、流程优化提供有效的数据参考。
4.  **信息不透明**：员工对团队整体的日志填写规范缺乏直观认知，不利于营造“及时复盘、透明协作”的文化氛围。

本项目的核心是解决上述管理痛点，通过数据化、可视化的方式，提升日志管理的效率与科学性。

### 1.2 项目目标
#### 业务目标（上线后3个月内）：
1.  将管理者查看团队日志填写情况的平均耗时从**人均每日15分钟降低至2分钟以内**。
2.  推动全公司正式员工的月度日志**整体填写率从当前的85%提升至95%以上**。
3.  为人力资源部的过程绩效管理提供至少**1个可量化的数据指标**（按时提交率）。

#### 产品目标：
1.  在管理后台新增“日志填写统计”报表模块，支持多维度、可定制的数据查询与展示。
2.  实现核心数据（填写率、按时提交率）的每日自动更新与计算，确保数据时效性。
3.  提供便捷的数据导出功能，支持线下分析与汇报。

#### 用户价值：
-   **对管理者（部门负责人、总监、HR）**：快速掌握团队日志填写概况，识别异常，辅助管理决策，提升管理效率。
-   **对高管**：宏观了解公司整体工作复盘文化的落地情况。
-   **对员工（间接价值）**：透明的数据公示可形成正向激励，促进按时填写习惯的养成。

### 1.3 受众范围
-   **产品与设计团队**：明确产品需求、设计原型与交互。
-   **开发团队（前端、后端）**：理解功能逻辑、接口定义、实现方案。
-   **测试团队**：依据本文档编写测试用例，进行功能与性能验证。
-   **运营与运维团队**：了解功能上线后的运营要点及系统维护需求。

### 1.4 术语表
| 术语 | 定义 |
| :--- | :--- |
| 工作日志/日报 | 员工每日/每周记录工作内容、进度、反思的线上记录。 |
| 填写率 | 在指定统计周期内，实际填写了日志的员工数 / 应填写日志的员工总数 * 100%。 |
| 按时提交率 | 在指定统计周期内，在规定的截止时间前提交日志的员工数 / 应填写日志的员工总数 * 100%。 |
| 平均填写时长 | 在指定统计周期内，所有员工从开始编辑到最终提交日志的平均耗时（分钟）。 |
| 统计周期 | 报表数据所覆盖的时间范围，如“2026-03-01至2026-03-31”。 |
| 正式员工 | 与公司签订正式劳动合同的员工，不含实习生、外包、顾问等。 |

## 2. 业务说明

### 2.1 用户角色
| 角色 | 身份 | 核心诉求与使用场景 |
| :--- | :--- | :--- |
| **系统管理员** | IT部门人员 | 配置报表基础参数，管理数据更新任务，处理数据异常。 |
| **人力资源专员** | HR部门人员 | 周期性（如每周、每月）统计全公司或特定部门日志填写情况，作为过程管理或文化建设的参考数据。 |
| **部门负责人** | 各业务部门总监/经理 | 每日/每周查看本部门员工日志填写情况，督促未填写或迟交的员工，了解团队工作复盘习惯。 |
| **公司高管** | CEO/VP等 | 不定期查看公司整体日志填写概况，宏观了解公司执行力与文化落地情况。 |

### 2.2 业务流程
#### 正常流程（以部门负责人查看上周部门报表为例）：
1.  **登录系统**：部门负责人登录PC端工作日报管理后台。
2.  **进入报表**：在主导航栏或工作台找到并点击“日志填写统计”入口。
3.  **设置查询条件**：
    -   维度选择：“部门”。
    -   部门选择：系统默认选中其管理的部门（可切换，但仅限其管辖范围）。
    -   统计周期：选择“上周”（系统预置选项，如“今日”、“本周”、“本月”、“上月”、“最近7天”、“最近30天”）。
    -   点击“查询”按钮。
4.  **查看报表**：系统展示该部门上周的日志填写统计概览（核心指标卡片）及员工明细列表。
5.  **数据下钻/导出**：
    -   点击某个较低的“填写率”，可查看具体是哪些员工未填写。
    -   点击“导出”按钮，将当前查询条件下的明细数据导出为Excel文件。
6.  **管理跟进**：根据报表数据，对未填写或迟交的员工进行提醒或沟通。

#### 异常流程：
-   **无数据情况**：当查询条件下无任何员工的日志记录时，页面清晰提示“当前查询条件下暂无数据”，并引导用户调整查询条件。
-   **权限不足**：当用户尝试查询其无权限访问的部门数据时，系统提示“您暂无权限查看该部门数据”。
-   **导出失败**：数据量过大或系统繁忙导致导出失败时，提示“导出失败，请稍后重试或联系系统管理员”。

### 2.3 使用场景
1.  **场景一（日常检查）**：销售部张经理每天上午10点，打开报表，选择维度“部门”，周期“昨日”，快速查看部门成员昨日的日志填写率。发现小李未填写，立即在企业微信上提醒。
2.  **场景二（周期复盘）**：人力资源部小王在每月5号，需统计上月全公司各部门的日志按时提交率，作为月度办公效率简报的一部分。她使用报表，选择维度“部门”，周期“上月”，导出Excel后，进行简要分析并制作图表。
3.  **场景三（问题排查）**：技术部陈总监发现近期项目复盘信息不全，怀疑与日志填写质量有关。他查看本部门“最近30天”的“平均填写时长”，发现平均时长不足3分钟，推断可能存在敷衍填写情况，计划在部门会议中强调日志填写规范。

## 3. 功能需求

### 3.1 功能全景
“日志填写统计”报表模块主要包含以下功能结构：
```
日志填写统计报表
├── 报表查询区（筛选条件设置）
├── 数据概览区（核心指标卡片展示）
├── 数据明细区（列表展示与排序）
└── 数据操作区（数据导出、帮助提示）
```

### 3.2 详细功能列表

#### 功能1：多维度数据查询
| 项目 | 说明 |
| :--- | :--- |
| **功能名称** | 统计报表查询筛选 |
| **功能说明** | 为用户提供灵活的条件来筛选和定位需要查看的统计报表数据。 |
| **操作流程** | 1. 用户进入报表页面，页面顶部显示查询区。<br>2. 用户依次选择或输入：统计维度、具体部门/员工、统计周期。<br>3. 点击“查询”按钮，下方展示区域刷新为对应数据。<br>4. 点击“重置”按钮，清空所有选项，恢复至默认状态（维度：部门；部门：用户所属第一个部门；周期：本周）。 |
| **输入/规则** | **统计维度**：单选，选项为“部门”、“员工”。<br>**部门选择**（当维度为“部门”时）：树形下拉选择器，仅展示用户有权限查看的部门。支持单选。<br>**员工选择**（当维度为“员工”时）：支持按姓名/工号搜索选择，可多选。默认展示用户所属部门下的员工。<br>**统计周期**：提供快捷选项（今日、昨日、本周、上周、本月、上月、最近7天、最近30天）和自定义日期范围选择器。选择自定义时，可指定开始日期与结束日期，间隔最长不超过365天。 |
| **输出** | 触发下方“数据概览区”和“数据明细区”的刷新，展示符合条件的数据。 |

#### 功能2：核心指标数据展示
| 项目 | 说明 |
| :--- | :--- |
| **功能名称** | 统计概览卡片展示 |
| **功能说明** | 以卡片形式突出展示查询条件下的核心统计指标，让用户快速把握整体情况。 |
| **操作流程** | 系统自动根据查询条件计算并展示，无需用户操作。 |
| **输入/规则** | 接收“功能1”的查询条件作为输入。 |
| **输出/规则** | 展示3张指标卡片：<br>1.  **员工填写率**：`(实际填写日志的员工数 / 应填写员工总数) * 100%`。应填写员工总数指在统计周期内，处于在职状态的正式员工数。<br>2.  **按时提交率**：`(规定截止时间前提交日志的员工数 / 应填写员工总数) * 100%`。规定截止时间由系统配置，默认假设为当日23:59:59。<br>3.  **平均填写时长**：`∑(每位员工单次日志提交时间戳 - 开始编辑时间戳) / 总填写次数`。单位：分钟，保留一位小数。若无可计算的数据，显示“--”。<br>**数据更新**：所有指标数据于每日凌晨2:00通过定时任务计算更新（T+1）。 |

#### 功能3：明细数据列表与排序
| 项目 | 说明 |
| :--- | :--- |
| **功能名称** | 统计明细列表 |
| **功能说明** | 以表格形式展示明细数据，支持排序，方便用户查看具体人员或部门的数据。 |
| **操作流程** | 1. 用户可在列表表头点击特定列（如“填写率”、“按时提交次数”）进行升序/降序排序。<br>2. 当维度为“员工”时，点击某员工姓名，可跳转至该员工的个人日志历史页面（假设此页面已存在）。 |
| **输入/规则** | 接收“功能1”的查询条件作为输入。 |
| **输出/规则** | 展示表格，列字段根据维度不同略有差异：<br>- **部门维度**：部门名称、员工总数、已填写人数、填写率、按时提交人数、按时提交率、平均填写时长（部门平均值）。<br>- **员工维度**：员工姓名/工号、所属部门、应填写天数、已填写天数、填写率、按时提交天数、按时提交率、平均填写时长（个人平均值）。<br>列表默认按“部门名称”或“员工姓名”拼音升序排列，分页显示，每页20条。 |

#### 功能4：数据导出
| 项目 | 说明 |
| :--- | :--- |
| **功能名称** | 报表数据导出 |
| **功能说明** | 将当前查询条件下“数据明细区”的列表数据导出为Excel文件，供线下留存、分析或汇报使用。 |
| **操作流程** | 1. 用户在查询结果页面，点击“导出Excel”按钮。<br>2. 系统开始准备数据，页面显示“正在生成，请稍候...”提示。<br>3. 生成完成后，浏览器自动下载文件。 |
| **输入/规则** | 当前页面所有查询条件。 |
| **输出/规则** | 导出的Excel文件命名规则：`日志填写统计_维度_部门名/员工名_统计周期_导出时间.xlsx`，例如：`日志填写统计_部门_市场部_20260301-20260331_20260401103025.xlsx`。<br>文件内容与当前页面“数据明细区”的表格列完全一致。 |

### 3.3 页面说明（报表主页面）
**页面路径**：管理后台 -> 工作日报 -> 日志填写统计
**页面布局**：自上而下分为查询筛选区、数据概览区、数据明细区。
**核心交互**：
1.  查询条件变更后，必须点击“查询”按钮才会刷新下方数据。
2.  “重置”按钮仅重置查询条件，不刷新下方数据，需再次点击“查询”。
3.  指标卡片和数据列表需有加载状态提示。
4.  导出文件较大时，需有进度提示或“后台导出，完成后通知”的机制（根据技术评估选择方案，此为假设）。

### 3.4 权限说明
-   **功能权限**：
    -   “日志填写统计”报表菜单入口：仅对**部门负责人及以上管理者**、**人力资源专员**、**系统管理员**可见。
-   **数据权限**：
    -   **部门负责人**：仅可查看和管理**其直接下属及所在部门**的数据。在部门选择器中，只能看到自己管理的部门树。
    -   **人力资源专员**：可查看**全公司所有部门**的数据。
    -   **系统管理员**：拥有全部数据查看权限，主要用于系统维护。
    -   **公司高管**：根据其管理的业务范围，在系统中配置相应的数据权限（可查看所辖业务线所有部门数据），**此需求假设权限系统已支持此配置，本次不开发新权限配置功能**。
-   **数据范围**：所有统计仅针对“正式员工”，需从HR系统同步的员工主数据中准确区分“正式员工”与“实习生”等类型。

## 4. 非功能性需求

### 4.1 性能需求
-   **页面响应时间**：在常规网络条件下，查询条件提交后，页面数据应在**3秒内**完成加载并展示。
-   **数据导出**：导出1万条记录以内的数据，生成Excel文件的时间应**不超过30秒**。
-   **数据处理时效**：每日凌晨的定时统计任务，应在**1小时内**完成全公司数据的计算与更新。
-   **并发量**：支持至少50个管理用户同时在线进行报表查询操作。

### 4.2 兼容性需求
-   **浏览器**：支持Chrome (80+)、Edge (Chromium, 80+)、Firefox (75+) 的最新两个稳定版本。确保在1920*1080及以上分辨率下显示正常。
-   **设备**：仅适配PC端Web浏览器，无需适配移动端。

### 4.3 安全性需求
-   **权限控制**：前后端均需进行严格的权限校验，防止越权查询数据。
-   **数据脱敏**：导出文件中的员工敏感信息（如手机号、身份证号，若存在）需进行脱敏处理（本次需求明细数据不包含此类信息，作为通用原则提出）。
-   **防攻击**：对查询和导出接口进行频率限制，防止恶意刷取数据。

### 4.4 可用性需求
-   **系统可用性**：报表模块整体可用性不低于99.5%。
-   **容错与提示**：
    -   查询无数据时，给予明确提示。
    -   网络异常或请求失败时，有统一友好的错误提示，并提示用户重试。
    -   日期选择不合规（如结束日期早于开始日期）时，在点击查询前给予即时校验提示。
-   **帮助与引导**：在报表页面合适位置（如查询区附近）提供“？”图标，点击可查看关于各指标计算规则的简要说明。
