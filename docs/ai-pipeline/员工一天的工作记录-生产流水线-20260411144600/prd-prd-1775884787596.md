## **1. 文档概述**

### **1.1 文档背景**
目前，公司项目成员的每日工作记录分散在各类即时通讯工具、个人笔记或简单的Excel表格中，导致：
1.  **项目管理困难**：项目经理难以实时、准确地掌握各项目实际工时投入，导致项目成本核算不清、资源调配缺乏依据。
2.  **数据汇总低效**：财务与运营部门每月需耗费大量人力手工收集、整理和统计工时数据，过程易出错，时效性差。
3.  **员工体验不佳**：员工记录工作内容方式不统一，且与项目管理系统脱节，存在重复劳动，常因遗忘导致日报补写，影响数据准确性。

为解决以上问题，亟需开发一个统一的、轻量化的“员工工作记录”功能，旨在将员工每日工作记录行为线上化、结构化，并与现有项目管理系统打通，实现工时数据的自动归集与分析。

### **1.2 项目目标**
*注：原始需求中未提供量化指标，此处根据行业经验进行合理补充。*
1.  **业务目标**：
    *   上线后3个月内，实现公司90%以上项目成员使用该系统记录每日工时。
    *   将财务部门月度工时统计与核算的人力耗时降低70%。
    *   为项目成本分析提供准确数据基础，支撑未来项目利润率提升5%的决策分析。
2.  **产品目标**：
    *   打造一个员工平均每日填写耗时低于3分钟、次日留存率高于85%的轻量化记录工具。
    *   实现与公司现有项目管理系统（PM系统）的数据双向同步，确保项目信息一致性。
    *   生成可视化工时报告，支持按日、周、月、项目、部门等多维度查看。
3.  **用户价值**：
    *   **对员工**：提供便捷、统一的记录入口，简化日报流程，通过关联项目自动归集个人工作成果。
    *   **对项目经理**：实时掌握项目成员工时投入，精准评估项目进度与资源消耗，辅助管理决策。
    *   **对管理层/财务**：自动获取准确的工时数据，用于成本核算、效率分析和资源规划。

### **1.3 受众范围**
*   **产品与设计团队**：用于理解需求，进行产品设计与交互设计。
*   **研发团队（前端、后端、移动端）**：用于系统开发、接口联调的详细依据。
*   **测试团队**：用于编写测试用例，执行功能与非功能测试。
*   **运维团队**：用于了解系统架构，准备部署与监控方案。
*   **运营与业务方**：用于了解产品最终形态，准备上线后的培训与推广材料。

## **2. 业务说明**

### **2.1 用户角色**
| 角色 | 核心诉求 | 使用场景 |
| :--- | :--- | :--- |
| **项目成员** (工程师、设计师等) | 快速、便捷地记录每日工作，减少操作负担；清晰看到个人工时汇总。 | 每日下班前填写日报；中途暂存；查看个人历史记录。 |
| **项目经理** | 实时查看所负责项目下所有成员的工时投入；监控项目整体人力成本。 | 查看项目维度工时报告；审批异常工时（*假设未来扩展功能*）。 |
| **部门负责人/管理层** | 宏观掌握部门或公司整体工时分布与效率情况。 | 查看部门/公司维度工时汇总与分析报告。 |
| **系统管理员** | 管理项目基础数据、用户权限、系统配置。 | 同步PM系统项目列表；配置提醒规则；管理用户角色。 |

### **2.2 业务流程**
**2.2.1 正常流程（员工提交日报）**
1.  **触发**：员工在每日下班前（或收到系统提醒后）打开Web端或移动端应用。
2.  **进入记录页**：点击“填写今日工作”或类似入口。
3.  **添加任务**：
    a. 选择或搜索项目：从下拉列表（数据源来自PM系统）中选择今日工作的项目，支持关键词搜索。若项目不存在，可触发“申请新增项目”流程（*需跳转至PM系统或走审批流，此为假设*）。
    b. 填写任务描述：简明描述具体工作内容（如：“开发用户登录模块后端接口”）。
    c. 填写工时：输入该任务花费的小时数（支持0.5小时为最小单位）。
    d. 点击“添加”按钮，该条任务记录显示在下方列表中。
4.  **重复添加**：重复步骤3，添加当日所有工作任务。
5.  **暂存或提交**：
    *   点击“暂存”：当前记录保存为草稿，可随时继续编辑。系统不进行工时统计。
    *   点击“提交”：系统校验总工时是否大于0且小于24小时。校验通过后，记录状态变为“已提交”，数据锁定不可编辑（*假设规则*），并同步至PM系统及后台数据库用于报告生成。员工收到提交成功提示。
6.  **结束**：流程结束。员工可查看“已提交”的历史记录。

**2.2.2 异常流程**
*   **异常1：未选择项目提交**
    *   流程：用户未选择项目即点击“提交”。
    *   系统处理：前端进行校验，弹出提示：“请为每项任务关联所属项目”，并高亮未选择项目的任务行。
*   **异常2：总工时异常**
    *   流程：用户提交时，当日总工时=0或>24小时。
    *   系统处理：提交时校验，提示：“总工时需大于0且小于等于24小时，请检查输入”。
*   **异常3：网络中断**
    *   流程：提交或暂存过程中网络断开。
    *   系统处理：操作失败，提示：“网络连接失败，请检查网络后重试”。对于提交操作，建议客户端自动缓存数据，待网络恢复后提示用户重新提交。

### **2.3 使用场景**
1.  **场景一：工程师小A记录开发工作**
    *   时间：周三下午17:50。
    *   地点：工位上。
    *   行为：小A收到企业微信提醒，打开PC端页面，快速选择项目“CRM系统重构”，添加三条任务：“接口开发-用户模块（3.5h）”、“Code Review（1h）”、“技术方案讨论（0.5h）”，总工时5小时，点击提交，用时2分钟。
2.  **场景二：设计师小B在外出差记录**
    *   时间：周五返程高铁上。
    *   地点：移动端。
    *   行为：小B在手机App上，利用项目搜索功能快速找到“官网视觉升级”项目，记录当天与客户沟通和修改设计稿的工作，分两次任务记录，并利用暂存功能先保存，下车后确认无误再提交。
3.  **场景三：项目经理老王查看项目周报**
    *   时间：周一上午。
    *   地点：办公室。
    *   行为：老王登录系统，进入“工时报告”模块，选择“项目视图”，筛选“XX智慧园区项目”及上周时间范围，系统展示出该周项目总工时、各成员工时分布饼图及每日投入趋势图，用于评估项目进展和本周计划制定。

## **3. 功能需求**

### **3.1 功能全景**
```
员工工作记录系统
├── 核心记录模块
│   ├── 今日工作填写（增、删、改任务）
│   ├── 草稿暂存与续填
│   └── 历史记录查看
├── 项目管理模块（数据同步）
│   ├── 项目列表同步与展示
│   └── 项目搜索与筛选
├── 数据报告模块
│   ├── 个人工时视图（日/周/月）
│   ├── 项目工时视图（需权限）
│   └── 数据导出（.xlsx）
├── 系统辅助模块
│   ├── 提交提醒（站内信/集成办公软件）
│   ├── 用户指引
│   └── 系统设置（管理员）
└── 后台数据同步服务（与PM系统）
```

### **3.2 详细功能列表**
**3.2.1 功能：今日工作记录**
*   **功能说明**：员工创建、编辑和提交当日工作记录的核心页面。
*   **操作流程**：见2.2.1业务流程。
*   **输入/输出规则**：
    *   **项目选择**：下拉列表，数据异步从PM系统同步。支持输入关键词搜索列表项。默认选项为“请选择项目”（不可提交）。
    *   **任务描述**：单行输入框，必填，字数限制1-200字符。
    *   **工时**：数字输入框，必填，允许输入0.5的倍数（如0.5， 1， 1.5， 2），范围0.5-12小时（单条任务合理性约束）。支持小数点后一位。
    *   **任务列表**：展示已添加的任务，每条任务可单独删除或编辑（在提交前）。列表底部显示“当日总工时：X小时”。
    *   **按钮**：“添加任务”、“暂存草稿”、“提交日报”。提交后按钮变为“已提交，不可编辑”。

**3.2.2 功能：项目列表管理（数据同步）**
*   **功能说明**：后台服务与前端展示层，确保记录时使用的项目列表与PM系统权威数据一致。
*   **操作流程**：
    1.  系统管理员在后台配置PM系统API同步地址与周期（如每日凌晨2点全量同步）。
    2.  同步服务定时拉取PM系统中“状态为进行中”的项目列表（字段至少包括：项目ID、项目名称、项目编码、负责人）。
    3.  员工在前端记录时，下拉列表加载本地缓存的同步后项目列表。
*   **输入/输出规则**：同步接口需遵循《现有PM系统API接口文档》。

**3.2.3 功能：可视化报告**
*   **功能说明**：为不同角色提供多维度工时数据图表。
*   **操作流程**：用户进入“报告”页面，选择视图类型（个人/项目）、时间范围（今日、本周、本月、自定义），系统渲染对应图表。
*   **输入/输出规则**：
    *   **个人视图**：折线图（每日总工时趋势）、柱状图（各项目工时分布），仅可查看自己的数据。
    *   **项目视图**（需项目经理及以上权限）：饼图（项目内成员工时占比）、柱状图（项目每日总工时投入），仅可查看自己负责的项目数据。
    *   **数据导出**：支持将当前视图数据导出为Excel文件。

### **3.3 页面说明**
**页面：工作记录主页面 (Web & Mobile)**
*   **头部**：显示“填写今日工作 - [日期]”，提供“历史记录”入口。
*   **主体 - 任务添加区**：
    *   一行表单：`[项目选择下拉框] + [任务描述输入框] + [工时输入框] + [“添加”按钮]`。
    *   项目选择框旁有“刷新项目列表”图标（手动触发同步缓存）。
*   **主体 - 任务列表区**：
    *   表格形式展示已添加任务（移动端卡片式）。
    *   列：项目名称、任务描述、工时、操作（删除图标）。
    *   底部汇总行：显示总工时。
*   **底部操作区**：
    *   按钮：“暂存草稿”（次要按钮）、“提交日报”（主要按钮，蓝色）。
    *   提交后，按钮区变为灰色提示“今日工作已提交”。

## **4. 非功能性需求**

### **4.1 性能需求**
1.  **页面响应**：核心记录页面（Web）首屏加载时间 ≤ 2秒（在标准办公网络下）。
2.  **操作响应**：添加/删除任务、提交/暂存等操作，前端反馈时间 ≤ 1秒。
3.  **并发能力**：支持每日固定时间点（如下班前30分钟）约500名用户同时提交的并发压力。
4.  **数据时效**：从员工提交到报告数据可查询，延迟 ≤ 5分钟。

### **4.2 兼容性需求**
1.  **Web端**：支持Chrome 80+、Edge 80+、Safari 14+等主流浏览器最新两个稳定版本。
2.  **移动端**：
    *   iOS: 支持iOS 13及以上系统，适配iPhone主流尺寸。
    *   Android: 支持Android 10及以上系统，适配主流品牌机型。
    *   *假设：移动端以H5形式嵌入公司统一办公App或企业微信/钉钉，暂不开发独立Native App。*

### **4.3 安全性需求**
1.  **认证与授权**：集成公司统一SSO登录系统。所有功能访问需基于4.4权限说明进行严格校验。
2.  **数据安全**：传输过程使用HTTPS加密。敏感数据（如工时明细）在数据库中进行加密存储。
3.  **防攻击**：对输入框进行SQL注入、XSS脚本攻击过滤。
4.  **日志审计**：记录所有用户的关键操作（登录、提交、删除、导出）日志，便于追溯。

### **4.4 可用性需求**
1.  **系统可用性**：核心服务（记录、提交）月度可用性 ≥ 99.5%。
2.  **容错能力**：在PM系统接口暂时不可用时，前端项目列表可使用最近一次成功的缓存数据，并给出“数据同步中，可能不是最新”的弱提示，不影响员工记录。
3.  **异常提示**：所有用户操作失败必须有明确、友好的错误提示，指导用户下一步该怎么做。
4.  **无障碍设计**：Web端页面需满足基本的WCAG 2.1 AA标准，确保色盲、键盘操作等场景下的可用性。
