## **1. 文档概述**

### **1.1 文档背景**
随着天山公司业务规模扩大与项目复杂度提升，传统依赖Excel、邮件、会议沟通的运营计划管理模式已无法满足当前需求。主要业务痛点如下：
*   **计划编制效率低下**：项目启动时，项目经理需手动从零搭建计划，耗时耗力，且容易遗漏标准节点。
*   **计划调整僵化滞后**：当项目出现延期、资源变更时，计划调整困难。关键路径变化无法实时感知，导致风险预警滞后。
*   **计划信息不透明**：计划以静态文件形式流转，各岗位（如设计、工艺、生产）无法实时获取与自己相关的任务视图，协同效率低。
*   **管理决策缺乏依据**：缺乏对多项目计划执行情况的统一可视化监控，管理层难以快速掌握整体运营健康度。

本系统旨在构建一个**可视化、可协同、可动态调整**的运营计划管理中心，为核心生产制造流程赋能。

### **1.2 项目目标**
*   **业务目标（可量化）**：
    *   将新项目标准计划编制时长从平均3天缩短至30分钟以内。
    *   计划变更的同步与关键路径重算响应时间控制在5秒内。
    *   项目关键节点（KCP）的按时达成率提升15%。
    *   减少因计划信息不同步导致的内部沟通会议时长30%。
*   **产品目标**：
    *   打造一个覆盖“模板配置 -> 计划生成 -> 执行监控 -> 动态调整”全生命周期的计划管理工具。
    *   实现计划数据与组织人事数据的实时联动。
    *   提供直观的甘特图展示与关键路径分析能力。
*   **用户价值**：
    *   **项目经理**：快速生成专业计划，敏捷应对变化，清晰掌控关键路径与风险。
    *   **部门经理/资源经理**：清晰了解本岗位/部门资源负荷，便于资源调配。
    *   **执行人员**：明确个人任务、工期、前后置依赖，减少沟通成本。
    *   **管理层**：通过仪表盘全局监控多项目进度与健康状态，支撑决策。

### **1.3 受众范围**
*   **产品与设计团队**：作为产品功能定义、交互设计的核心输入。
*   **研发团队**：作为系统开发、数据库设计的依据。
*   **测试团队**：作为编写测试用例、执行测试验证的基准。
*   **实施与运维团队**：作为系统部署、配置和后期运维的指导。
*   **业务部门（运营、生产）**：作为需求确认和验收的参考。

### **1.4 术语表**
| 术语 | 全称/解释 |
| :--- | :--- |
| **KCP** | Key Control Point，关键控制点。项目中的里程碑式重要节点，通常需要评审或交付特定成果。 |
| **FS** | Finish-to-Start，完成-开始关系。前置任务完成后，后续任务才能开始。 |
| **SS** | Start-to-Start，开始-开始关系。前置任务开始后，后续任务才能开始。 |
| **FF** | Finish-to-Finish，完成-完成关系。前置任务完成后，后续任务才能完成。 |
| **SF** | Start-to-Finish，开始-完成关系。前置任务开始后，后续任务才能完成。 |
| **关键路径** | 项目中决定项目最早完成时间的一系列任务序列，该路径上的任何延迟都会导致整个项目延期。 |
| **甘特图** | 一种以条形图形式展示项目时间表的工具，横轴表示时间，纵轴表示任务，条形长度表示任务工期。 |

## **2. 业务说明**

### **2.1 用户角色**
| 角色 | 核心诉求 | 使用场景举例 |
| :--- | :--- | :--- |
| **系统管理员** | 维护系统基础数据，管理用户账号与权限。 | 为新员工创建账号并分配项目角色。 |
| **模板管理员** | 创建和维护公司级/部门级的计划模板，确保标准化。 | 根据新的工艺规范，更新“新产品试制”模板。 |
| **项目经理** | 高效制定项目计划，监控进度，处理变更，确保项目按时交付。 | 接到新订单后，快速生成项目计划并分配任务。 |
| **资源经理/部门主管** | 查看并协调本部门资源在多个项目中的负荷情况。 | 评估下个月设计工程师的工作负荷，进行资源平衡。 |
| **任务执行人** | 清晰了解自己的任务、要求、截止时间，反馈进度。 | 每天登录系统查看“我的任务”，更新任务状态。 |
| **高层管理者** | 宏观掌握多项目整体进展、资源利用率和关键风险。 | 每周通过管理驾驶舱查看所有在研项目的健康状态。 |

### **2.2 业务流程**
**2.2.1 主流程：计划创建与执行监控**
1.  **启动**：项目经理在系统中选择或搜索合适的“计划模板”。
2.  **生成**：项目经理输入项目基本信息（如项目名称、代码）和计划开始时间。系统根据模板自动生成初始计划甘特图。
3.  **资源配置**：项目经理为计划中每个关联了“岗位”的节点，从系统拉取的人员列表中选择具体的“执行人”。（*假设：人员数据通过接口从人事系统实时同步*）
4.  **发布**：项目经理确认计划无误后，点击“发布”。系统自动通知所有相关任务执行人。
5.  **执行与反馈**：执行人登录系统查看“我的任务”，更新任务状态（如：进行中、已完成、受阻）。系统自动更新甘特图进度。
6.  **监控**：项目经理通过甘特图实时监控进度，关键路径高亮显示。如有延迟，系统提供预警。

**2.2.2 变更流程：计划动态调整**
1.  **触发变更**：因需求变更、资源变动或任务延期，项目经理需要调整计划。
2.  **手动调整**：在甘特图上直接拖拽任务条调整时间，或编辑任务属性（工期、依赖关系）。
3.  **系统重算**：系统根据新的任务逻辑和工期，自动重新计算整个项目的关键路径和所有任务的起止时间。
4.  **影响分析**：系统提示本次调整对项目结束时间、关键路径及关联任务的影响。
5.  **确认与同步**：项目经理确认调整，系统再次通知受影响的任务执行人。

**2.2.3 异常流程**
*   **资源冲突**：当为任务分配人员时，若该人员在同一时间段已被其他项目占用，系统应给出明确冲突提示。
*   **依赖环**：当手动调整依赖关系形成循环依赖（A依赖B， B依赖C， C又依赖A）时，系统应禁止此操作并报错。
*   **数据同步失败**：当从人事系统同步组织架构和人员数据失败时，系统应记录日志并告警管理员，同时计划生成和资源配置功能应降级为使用本地缓存数据或手动输入模式。

### **2.3 使用场景**
1.  **场景一：新项目快速启动**
    > 销售部赢得一个新客户订单，要求3个月内交付。项目经理王工登录系统，在“模板库”中选择“标准产品交付”模板，输入项目开始日期为明天，系统瞬间生成了一个包含50多个任务的详细计划。王工只需为“结构设计”、“样机装配”等关键节点指派具体工程师，半小时内就完成了一份专业、可执行的项目计划并下发。

2.  **场景二：应对突发延期**
    > 负责“电路板调试”的李工程师因故请假，其负责的任务将延期3天。项目经理在甘特图上将该任务条向右拖拽3天。系统立即重新计算，显示**关键路径已改变**，原关键路径上的“整机测试”任务变为非关键，而“软件联调”任务变为新的关键任务，且项目总工期将延迟1天。项目经理据此立即与软件团队沟通预警。

3.  **场景三：管理层全局审视**
    > 运营总监张总在月度经营会上，打开系统的“管理驾驶舱”页面。视图上同时展示了公司所有正在执行的12个项目的甘特图缩影、整体进度百分比、以及“红灯”（严重延期）、“黄灯”（风险预警）、“绿灯”（正常）的项目计数。张总一眼就发现“XX特种设备项目”亮红灯，点击钻取查看详情，定位到是“外购件采购”节点受阻，随即在会上追问采购部负责人。

## **3. 功能需求**

### **3.1 功能全景**
系统整体功能架构分为四层：
*   **基础数据层**：用户/权限、组织/岗位/人员（对接）、项目基础信息。
*   **核心配置层**：计划模板管理，是系统灵活性的基石。
*   **核心应用层**：计划生成、计划调整、甘特图可视化、关键路径计算，是系统的价值核心。
*   **输出与集成层**：计划导出、报表、以及未来可能的与MES、OA等系统的数据接口。

### **3.2 详细功能列表**

#### **3.2.1 模块一：基础配置**
*   **功能1：用户与权限管理**
    *   **说明**：基于RBAC模型，管理系统访问权限。
    *   **流程**：管理员创建角色（如：项目经理、观察员），为角色分配功能权限（如：可编辑计划、仅查看计划）和数据权限（如：只能查看本部门项目），再将角色赋予相应用户。
*   **功能2：项目基础数据管理**
    *   **说明**：维护项目分类、项目状态等基础字典数据。
*   **功能3：组织与人员数据同步**
    *   **说明**：通过API接口定时/实时从人事系统同步公司组织架构、岗位和人员信息。提供手动触发同步和查看同步日志的功能。

#### **3.2.2 模块二：计划模板管理**
*   **功能1：模板库视图**
    *   **说明**：以列表或卡片形式展示所有模板，支持按类型、创建部门筛选。
*   **功能2：模板编辑**
    *   **说明**：核心功能，允许用户创建/编辑模板。
    *   **操作流程**：
        1.  创建新模板，填写模板名称、描述、适用项目类型。
        2.  在画布区通过“添加节点”按钮或拖拽方式创建任务节点。
        3.  编辑节点属性：**节点名称**、**标准工期**（单位：工作日）、**是否为KCP**（勾选框）、**关联岗位**（下拉选择，数据来自基础配置）。
        4.  建立节点依赖：在节点间拉取连线，并选择**依赖类型（FS/SS/FF/SF）**和**延时**（可正可负）。
        5.  保存模板。
*   **输入输出规则**：
    *   节点名称必填，最长50字符。
    *   标准工期为大于0的整数。
    *   一个模板必须至少有一个节点。
    *   依赖关系不能形成闭环。

#### **3.2.3 模块三：计划生成与管理（核心）**
*   **功能1：基于模板生成计划**
    *   **说明**：将模板实例化为具体项目计划。
    *   **流程**：选择模板 -> 输入项目信息及计划开始日期 -> 系统生成初始甘特图 -> 进入计划编辑页。
*   **功能2：甘特图可视化与交互**
    *   **说明**：系统核心交互界面。
    *   **页面元素**：
        *   左侧：任务树状列表（显示任务名称、工期、开始/结束时间、负责人、前置任务）。
        *   右侧：甘特图时间轴，任务以条形图显示。
        *   工具栏：缩放（日/周/月）、切换显示/隐藏关键路径、导出、全屏等按钮。
    *   **交互逻辑**：
        *   **关键路径高亮**：关键路径上的任务条以**红色粗边框**或特殊颜色（如橙色）显示。
        *   **拖拽调整**：可直接拖拽任务条的两端调整开始/结束时间（改变工期），或整体拖拽平移。
        *   **依赖关系调整**：可拖拽任务条之间的连线箭头，连接到新的前置/后续任务上。
        *   **右键菜单**：在任务条上右键，可弹出菜单进行“编辑详情”、“添加子任务”、“删除”、“设为KCP”等操作。
*   **功能3：计划动态调整与关键路径重算**
    *   **说明**：任何对任务时间或逻辑关系的修改，都会触发后台关键路径算法（CPM）的重新计算。
    *   **流程**：用户修改 -> 前端立即发送请求 -> 后端计算并返回新的时间表及关键路径 -> 前端更新甘特图渲染。
*   **功能4：资源（人员）分配**
    *   **说明**：在任务树状列表或任务详情弹窗中，为“关联岗位”的节点指派具体人员。
    *   **规则**：选择人员时，系统应调用该人员的“已分配任务日历”，若存在时间冲突，应给出明确视觉提示（如人名标红）。
*   **功能5：计划导出**
    *   **说明**：支持将当前视图下的计划导出为Excel文件。
    *   **输出要求**：Excel应包含任务层级、名称、开始/结束时间、工期、负责人、前置任务、是否关键路径等列。甘特图部分可以以条件格式模拟条形图。

## **4. 非功能性需求**

### **4.1 性能需求**
*   **页面响应**：常规页面（列表、表单）加载时间 ≤ 2秒。核心甘特图页面（含50个任务节点）首次加载及渲染完成时间 ≤ 3秒。
*   **操作响应**：甘特图拖拽调整、依赖关系修改等操作，前端反馈（如任务条移动）应即时（<100ms），后端重算关键路径并返回数据的整体响应时间 ≤ 3秒。
*   **并发支持**：系统需支持至少500名用户同时在线，100个用户并发进行核心计划编辑操作。
*   **数据容量**：单项目计划支持不少于1000个任务节点。系统整体支持存储至少5年的历史项目数据。

### **4.2 兼容性需求**
*   **浏览器**：必须完全兼容Chrome浏览器最新稳定版及最近两个大版本。确保在Edge、Firefox最新版上核心功能可用，UI样式允许有细微差异。
*   **屏幕分辨率**：适配主流桌面分辨率（1920x1080， 1366x768），在1280x720分辨率下保证核心功能可用。

### **4.3 安全性需求**
*   **访问安全**：采用HTTPS协议。用户会话具有超时机制（如30分钟无操作需重新登录）。
*   **权限控制**：所有后端API接口必须进行严格的角色权限校验，防止越权操作。
*   **数据安全**：用户密码等敏感信息需加密存储。操作日志需完整记录（何人、何时、对何数据、进行何种操作）。
*   **输入校验**：前后端均需对用户输入进行有效性、合法性校验，防范SQL注入和XSS攻击。

### **4.4 可用性需求**
*   **系统可用性**：要求达到99.5%以上（不含计划内维护时间）。
*   **容错与提示**：
    *   用户操作错误时（如删除前未确认），应有明确、友好的提示信息，告知错误原因。
    *   网络中断或后端服务异常时，前端应有统一且友好的错误提示页，并尝试自动重连。
    *   关键操作（如发布计划、删除任务）必须提供二次确认弹窗。
*   **易用性**：界面设计需符合工业软件用户习惯，布局清晰，重点突出。甘特图操作（拖拽、连线）需提供简要的新手引导或提示气泡。

## **5. 其他说明**

### **5.1 上线计划**
*   **Phase 1（MVP， 2025-10-27）**：核心功能上线。包含：基础配置、模板管理、计划生成、基础甘特图展示与调整、关键路径计算、人员分配、导出Excel。
*   **Phase 2（2026-01-27）**：增强协同与监控。包含：任务执行人进度反馈、站内通知/邮件提醒、基础报表/管理驾驶舱。
*   **Phase 3（2026-04-27， 正式上线）**：系统集成与优化。包含：与人事系统深度对接（同步在职状态、负荷计算）、性能优化、移动端轻量视图（H5）、高级报表。
*   **迭代节奏**：采用双周迭代模式，每个迭代周期明确开发、测试、演示功能范围。

### **5.2 运营建议**
1.  **种子用户培养**：上线前，邀请3 (5位资深项目经理作为种子用户，深度参与测试和反馈，并制定内部推广素材（如成功案例）。
2.  **分层培训**：为管理员、项目经理、执行人员分别制作操作手册和短视频教程，组织专场培训会。
3.  **制度保障**：建议公司发文，将“新项目计划必须通过系统编制和发布”纳入项目管理规范，与绩效考核适当挂钩。
4.  **持续收集反馈**：在系统内设立便捷的反馈入口，定期（每月）收集用户痛点，规划迭代优化。

### **5.3 风险评估与应对**
| 风险描述 | 可能性 | 影响程度 | 应对方案 |
| :--- | :---: | :---: | :--- |
| **与人事系统对接延迟或数据不一致** | 中 | 高 | **提前沟通**：与IT部门明确接口规范与排期。**降级方案**：一期支持手动维护人员名单，保证核心功能可用。 |
| **用户习惯抵触，尤其是资深项目经理** | 高 | 中 | **价值驱动**：通过种子用户示范，展示效率提升的量化效果。**降低门槛**：提供一键导入旧Excel计划的功能（可作为二期需求）。**高层推动**：争取管理层在会议上的公开支持和使用。 |
| **复杂项目计划导致前端性能瓶颈** | 中 | 高 | **技术预研**：在技术选型阶段，重点评估前端甘特图组件的性能。**设计优化**：采用虚拟滚动、分步加载、懒加载等技术。设定性能红线并严格测试。 |
