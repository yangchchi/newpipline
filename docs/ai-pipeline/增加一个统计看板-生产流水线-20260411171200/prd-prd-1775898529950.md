# 工作日志统计看板PRD文档

## 1. 文档概述

### 1.1 文档背景
**项目发起原因：**
目前公司内部工作日志模块已上线运行X个月，日志填写已成为员工日常工作流程的一部分。但管理层（尤其是部门负责人与管理员）缺乏有效的数据化工具来监控日志填写情况，导致管理效率低下，无法及时发现和解决问题。

**业务痛点：**
1.  **管理盲区：** 部门负责人无法直观了解本部门员工的日志填写整体情况（如谁未提交、谁经常延迟），只能通过人工逐一检查或询问，耗时耗力。
2.  **决策无据：** 管理员无法从全局视角掌握公司日志填写健康度，难以评估此项制度的执行效果，也无法基于数据优化管理策略。
3.  **效率低下：** 当前统计方式依赖人工导出Excel后手动处理，数据滞后且易出错，无法满足实时或准实时监控的需求。
4.  **体验不佳：** 管理层在日志管理后台缺乏一个直观、聚合的数据视图，需要多次点击跳转才能获取零散信息。

**核心解决问题：**
为管理层提供一个自动化、可视化、权限隔离的数据看板，实现对工作日志填写情况的实时监控与量化管理，提升管理效率和决策科学性。

### 1.2 项目目标
**业务目标（可量化）：**
1.  上线后1个月内，管理员查看全公司日志填写整体情况的时间从平均30分钟/次缩短至1分钟以内。
2.  上线后1个季度内，通过看板预警与干预，全公司日志提交率（按时提交的日志数/应提交日志总数）从当前的85%提升至95%。
3.  提升部门负责人对团队日志管理的效率，预计节省其每周约2小时的统计与核对时间。

**产品目标：**
1.  **功能完整性：** 交付一个包含部门/员工双维度、提交率/及时率/平均填写时长核心指标、支持日/周/月筛选、可视化图表展示的统计看板。
2.  **数据准确性：** 确保看板数据与底层日志数据100%一致，数据更新延迟不超过1个自然日。
3.  **用户体验：** 看板加载时间小于3秒，界面清晰直观，操作简单，符合管理后台设计规范。
4.  **权限安全：** 100%实现数据权限隔离，确保用户仅能查看权限范围内的数据。

**用户价值：**
1.  **部门负责人：** 快速掌握团队日志填写状况，识别“问题员工”，进行针对性管理，提升团队执行力。
2.  **系统管理员/公司管理层：** 全局把控日志制度执行效果，通过趋势分析评估政策影响，为管理优化提供数据支持。
3.  **（间接）全体员工：** 更透明、公平的管理环境，督促形成良好的日志填写习惯。

## 2. 业务说明

### 2.1 用户角色
| 角色 | 核心诉求 | 使用场景 |
| :--- | :--- | :--- |
| **部门负责人** | 1. 快速了解本部门整体及个人日志填写情况。<br>2. 及时发现未提交、经常延迟提交的员工作出提醒。<br>3. 评估团队时间投入分布（通过填写时长侧面反映）。 | 每日晨会前查看昨日情况；每周一查看上周总结；每月初查看上月情况。 |
| **系统管理员/公司高管** | 1. 掌握全公司日志填写整体健康度与趋势。<br>2. 对比不同部门间的执行差异。<br>3. 为是否调整日志填写政策提供数据依据。 | 定期（如每周）查看公司级报告；在推行新政策后观察数据变化。 |
| **（潜在）HR/运营** | 1. 将日志填写数据作为员工执行力或习惯的参考指标之一。 | 在进行团队或个人绩效辅助评估时。 |

### 2.2 业务流程
**正常流程：**
1.  **数据生成：** 员工每日填写/提交工作日志，系统记录相关行为数据（提交时间、编辑时长等）。
2.  **定时聚合：** 系统每日凌晨01:00（假设）启动定时任务，基于前一天的日志数据，按预设规则（部门、员工、日、周、月维度）计算提交率、及时率、平均填写时长等指标。
3.  **数据存储：** 计算结果存储于统计汇总表中，供看板实时查询。
4.  **用户访问：** 用户（部门负责人/管理员）登录日志管理后台，首页显眼位置展示统计看板。
5.  **权限校验：** 用户访问看板时，系统根据其角色和所属部门，动态过滤数据查询范围。
6.  **数据展示：** 看板加载用户权限内的聚合数据，并以指标卡和可视化图表形式呈现。
7.  **交互筛选：** 用户可通过看板上方的筛选器，选择不同的统计维度（部门/员工）、时间粒度（日/周/月）以及具体时间范围，图表数据随之动态刷新。

**异常流程：**
1.  **数据延迟/异常：**
    - **场景：** 定时任务执行失败或数据源异常。
    - **处理：** 看板显示“数据正在更新中”提示，并展示最近一次成功更新的数据及时间戳。系统应发出告警通知运维人员。
2.  **无权限访问：**
    - **场景：** 普通员工误入或有权限用户访问他人数据。
    - **处理：** 系统拦截请求，看板区域显示“暂无查看权限”或重定向至无权限提示页。
3.  **无数据情况：**
    - **场景：** 新部门或新员工在选定时间范围内无任何日志数据。
    - **处理：** 图表区域显示“暂无数据”的占位图或友好提示文字。

### 2.3 使用场景
**场景一：部门负责人张经理的每日晨检**
- **何时：** 工作日上午9:15，开始每日站会之前。
- **何人：** 研发部负责人张经理。
- **何事：** 张经理打开日志管理后台，首页看板自动展示“昨日”数据。他快速扫视部门整体提交率（98%）和及时率（95%），发现均达标。柱状图显示有1名员工未提交，点击该柱状图下钻，看到是员工李四。他记下名字，准备在站会上提醒。
- **价值：** 1分钟内完成团队日志合规性检查，实现精准管理。

**场景二：管理员王总查看月度执行报告**
- **何时：** 每月5号，准备月度管理会议材料时。
- **何人：** 公司系统管理员王总。
- **何事：** 王总进入看板，将筛选条件调整为“月度”、“上月”（2026年3月），维度选择“部门”。趋势图显示近三个月全公司提交率稳步上升。部门对比柱状图显示市场部的及时率明显偏低。王总将图表截图，准备在管理会议上询问市场部原因，并讨论优化方案。
- **价值：** 基于客观数据发现管理短板，驱动跨部门协同改进。

**场景三：新政策效果评估**
- **何时：** 公司推行“日志需在当日下班前提交”新规后的第二周。
- **何人：** 运营总监兼管理员赵总。
- **何事：** 赵总将看板时间筛选为“周”，对比新规执行前后两周的“及时率”趋势线。发现整体及时率有显著提升，但技术部提升不明显。他下钻查看技术部详情，发现部分员工平均填写时长过长。他判断可能需要对新规进行针对性培训或优化工具。
- **价值：** 量化评估政策影响，指导后续优化方向。

## 3. 功能需求

### 3.1 功能全景
工作日志统计看板作为日志管理后台的一个新增核心功能模块，其架构如下：
```
工作日志管理后台
├── 日志列表管理 (已有)
├── 模板管理 (已有)
├── **统计看板 (新增)**
│   ├── 1. 全局筛选区
│   ├── 2. 核心指标卡
│   ├── 3. 数据可视化图表区
│   │   ├── 3.1 提交情况趋势图
│   │   ├── 3.2 部门/员工对比柱状图
│   │   └── 3.3 填写时长分布图 (二期)
│   └── 4. 数据详情下钻 (二期)
└── 系统设置 (已有)
```

### 3.2 详细功能列表

#### 功能1：全局筛选器
- **功能说明：** 用户通过此区域控制看板数据的统计维度与时间范围。
- **操作流程：**
    1.  用户进入看板页面。
    2.  页面顶部展示筛选控件。
    3.  用户选择【维度】（部门/员工）、【时间粒度】（日/周/月）、【时间范围】（如日期选择器）。
    4.  点击“查询”或条件变更后自动触发看板数据刷新。
- **输入输出规则：**
    - **输入：** 维度选择、时间粒度选择、具体日期/周/月选择。
    - **输出：** 向后台传递筛选参数，获取并渲染新的看板数据。
    - **规则：**
        - 【维度】为“部门”时，图表以部门为维度展示；为“员工”时，展示当前权限范围内所有员工的聚合或列表数据（**假设一期默认展示Top20员工，支持搜索**）。
        - 【时间粒度】切换时，【时间范围】选择器自动切换为对应类型（日选择器、周选择器、月选择器）。
        - 默认进入时，维度为“部门”，时间粒度为“日”，时间范围为“昨日”。

#### 功能2：核心指标卡 (KPI Cards)
- **功能说明：** 以突出数字的形式展示当前筛选条件下的核心统计结果。
- **操作流程：** 自动展示，无需操作。
- **输入输出规则：**
    - **展示内容（三个指标卡）：**
        1.  **日志提交率：** `[已提交数] / [应提交总数]`，百分比形式，保留一位小数（如96.5%）。
        2.  **日志及时率：** `[及时提交数] / [已提交总数]`，百分比形式，保留一位小数。
        3.  **平均填写时长：** `Σ(单次日志填写时长) / [已提交总数]`，以分钟为单位，取整数展示。超过60分钟可考虑换算为“X小时Y分钟”格式。
    - **规则：** 指标卡数值随全局筛选器条件变化而动态更新。

#### 功能3：数据可视化图表区
- **功能说明：** 通过图形化方式展示数据趋势与对比。
- **操作流程：** 图表根据筛选条件自动生成。用户可悬停在图表数据点上查看具体数值。

**3.3.1 提交情况趋势图**
- **图表类型：** 双线折线图。
- **X轴：** 时间（根据时间粒度显示为具体日期、周次或月份）。
- **Y轴：** 比率（0%-100%）。
- **数据线：** 两条，分别代表“提交率”和“及时率”在选定时间范围内的变化趋势。
- **交互：** 鼠标悬停显示对应时间点的具体数值。

**3.3.2 部门/员工对比柱状图**
- **图表类型：** 分组柱状图或堆叠柱状图（视展示清晰度定）。
- **X轴：** 部门名称或员工姓名（受维度筛选控制）。
- **Y轴：** 数量或比率。
- **数据系列：**
    - 当维度为“部门”时：展示各部门的“应提交数”、“已提交数”、“及时提交数”。（可用堆叠柱状图表示构成）。
    - 当维度为“员工”时：展示Top N员工的“提交率”和“及时率”（两个并排的柱状）。
1  **规则：** 员工维度下，默认按提交率从低到高排序展示Top 20，支持通过搜索框查找特定员工。

### 3.3 页面说明
**页面位置：** 日志管理后台首页，首屏核心区域。
**页面布局（草图描述）：**
```
+---------------------------------------------------+
|  [工作日志] [统计看板] [模板管理] ... (导航栏)     |

+---------------------------------------------------+
|  | 提交率: 96.5% | | 及时率: 94.2% | | 平均时长: 25min | (指标卡)|
+---------------------------------------------------+
|                                                    |
|  **提交情况趋势**          **部门提交对比**        |
|  | 折线图区域   |          | 柱状图区域   |        |

```
**交互逻辑：**
1.  筛选器联动：切换“时间粒度”，“时间”选择器组件类型联动变化。
2.  图表响应筛选：点击“查询”或筛选条件变化后，指标卡与两个图表区域异步加载新数据并重绘。
3.  默认状态：页面加载后自动以默认条件（部门、昨日）查询并渲染数据。

### 3.4 权限说明
| 功能点/数据项 | 部门负责人 | 系统管理员 | 普通员工 |
| :--- | :--- | :--- | :--- |
| **访问统计看板入口** | ✅ 可见且可访问 | ✅ 可见且可访问 | ❌ 不可见/无权限 |
| **全局筛选器** | ✅ 可用，但“维度”为员工时，仅能筛选本部门员工 | ✅ 完全可用 | - |
| **核心指标卡** | ✅ 仅展示本部门数据 | ✅ 展示全公司数据 | - |
| **趋势图** | ✅ 仅展示本部门趋势 | ✅ 展示全公司趋势 | - |
| **对比柱状图** | ✅ 仅对比本部门内员工或子部门（若有） | ✅ 可对比全公司所有部门 | - |
| **下钻查看明细（二期）** | ✅ 仅能下钻至本部门员工个人日志列表 | ✅ 可下钻至任何部门或员工 | - |

**权限实现逻辑：**
- 后端所有数据查询接口必须增加权限校验参数。
- 部门负责人：查询时自动附加其管辖的部门ID列表作为过滤条件。
- 系统管理员：查询时不附加部门过滤条件，或附加全公司部门ID列表。

## 4. 非功能性需求

### 4.1 性能需求
- **页面加载时间：** 在标准网络环境下（4G），看板首页（包含默认数据）完全加载并渲染完成的时间应 ≤ 3秒。
- **数据查询响应：** 切换筛选条件后，图表数据重新加载和渲染的响应时间应 ≤ 2秒。
- **并发支持：** 支持至少50个管理员/部门负责人同时在线操作看板。
- **数据处理时效：** 看板展示的统计数据，其更新延迟不超过1个自然日。即T日看到的数据至少包含T-1日23:59:59之前提交的日志。

### 4.2 兼容性需求
- **浏览器：** 支持Chrome 85+、Firefox 80+、Safari 14+、Edge (Chromium) 85+等主流浏览器的最新两个稳定版本。
- **屏幕分辨率：** 适配主流桌面分辨率（1920*1080, 1366*768），确保在笔记本小屏上核心内容可见，无需水平滚动。
- **（移动端暂非必须）**

### 4.3 安全性需求
- **权限控制：** 必须严格执行3.4中定义的权限规则。所有数据接口需进行服务端权限验证，防止前端绕过。
- **数据安全：** 看板数据为聚合数据，不直接暴露敏感日志内容。未来下钻功能需进行二次权限校验。
- **防攻击：** 对数据查询接口进行限流，防止恶意刷接口导致数据库压力过大。
- **日志记录：** 记录管理员和部门负责人访问看板的关键操作日志（如查询时间、参数），以备审计。

### 4.4 可用性需求
- **系统可用性：** 看板及相关服务目标可用性 ≥ 99.5%。
- **容错能力：** 前端图表组件加载失败时，应显示友好的错误提示（如“图表加载失败，请刷新重试”），而非空白或JS错误。
- **数据异常提示：** 当数据源异常或定时任务失败时，看板应有明确提示（如“数据更新延迟，最后更新于XX:XX”）。
- **空状态：** 当查询结果无数据时，图表区域应展示“暂无数据”的占位图或提示文案，引导用户调整筛选条件。
