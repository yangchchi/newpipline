## 1. 文档概述

### 1.1 文档背景
天山公司作为一家典型的生产制造企业，其运营管理面临以下核心痛点：
1.  **计划制定效率低下**：项目计划严重依赖项目经理个人经验，通过Excel手动编制，耗时耗力，且易出错，难以快速响应新的生产项目需求。
2.  **计划调整僵化**：当项目进度发生变更（如物料延迟、设备故障、人员变动）时，依赖手工调整Excel中的任务时间与依赖关系，关键路径需要人工重新计算，响应慢且不准确，导致资源调配滞后。
3.  **协同与可视化不足**：计划以文档形式分发，各岗位（如设计、采购、生产）对整体进度和自身任务节点缺乏直观感知，信息同步不及时，容易造成工序等待或资源冲突。
4.  **经验难以沉淀**：优秀的项目计划模板无法有效固化并复用，组织过程资产积累不足，新项目经理上手成本高。

为解决以上问题，亟需建设一个专业的、可视化的运营计划管理工具，以提升企业项目管理的精细化、敏捷化和标准化水平。

### 1.2 项目目标
**业务目标（可量化）：**
- 项目计划平均编制时间缩短 **40%**（从当前平均2天/项目缩短至1.2天/项目）。
- 计划调整后关键路径重新计算的准确率达到 **100%**，调整响应时间控制在 **5分钟** 内。
- 试点部门（如生产部、项目部）关键节点（KCP）按时达成率提升 **15%**。
- 系统上线6个月内，沉淀有效项目计划模板不少于 **10个**。

**产品目标：**
- 交付一个稳定、易用的Web端运营计划管理系统（PMS），支持计划模板化生成、可视化调整与协同跟踪。
- 实现与现有组织架构、人事系统的数据打通，确保基础数据一致性。
- 提供甘特图、关键路径法（CPM）等专业项目管理视图，满足生产制造场景需求。

**用户价值：**
- **项目经理**：快速制定和调整计划，精准把控项目关键路径，提升管理效率与决策质量。
- **部门主管/岗位人员**：清晰了解个人任务、前后置依赖及项目整体进度，减少沟通成本，增强协同。
- **公司管理层**：可视化监控多项目整体进展与资源负荷，为战略决策提供数据支持。

### 1.3 受众范围
- **产品与设计团队**：作为产品设计、原型制作的依据。
- **开发团队（前端、后端）**：作为系统开发、接口定义的指导文档。
- **测试团队**：作为编写测试用例、进行功能与集成测试的依据。
- **运维团队**：了解系统架构与非功能性需求，为部署和监控做准备。
- **业务方（天山公司运营部、生产部）**：确认需求范围与业务逻辑的准确性。

### 1.4 术语表
| 术语 | 全称/解释 |
| :--- | :--- |
| **PMS** | 运营计划管理系统（Project Management System） |
| **KCP** | 关键控制点（Key Control Point），指项目中必须重点监控和确保完成的里程碑式节点。 |
| **甘特图** | 一种通过条形图显示项目进度、任务及其时间安排的图表。 |
| **关键路径** | 项目中时间最长的任务序列，决定了项目的最短完成时间。关键路径上的任何延迟都会导致项目整体延迟。 |
| **节点依赖** | 任务之间的逻辑关系，决定了一个任务开始或结束如何受另一个任务影响。 |
| **FS** | 完成-开始（Finish-to-Start）：前置任务完成后，后续任务才能开始。 |
| **SS** | 开始-开始（Start-to-Start）：前置任务开始后，后续任务才能开始。 |
| **FF** | 完成-完成（Finish-to-Finish）：前置任务完成后，后续任务才能完成。 |
| **SF** | 开始-完成（Start-to-Finish）：前置任务开始后，后续任务才能完成。（较少用） |
| **岗位** | 组织内的职能角色，如“工艺工程师”、“生产班长”。 |
| **人员** | 具体的员工，隶属于某个岗位。 |

## 2. 业务说明

### 2.1 用户角色
| 角色 | 核心诉求 | 使用频率 |
| :--- | :--- | :--- |
| **系统管理员** | 1. 维护系统基础数据（用户、角色、权限）<br>2. 配置与外部系统的数据同步<br>3. 监控系统运行状态，处理异常 | 低 |
| **计划管理员（高级用户）** | 1. 创建和维护公司级/部门级的计划模板<br>2. 管理项目基础数据（岗位）<br>3. 可能需要查看所有项目计划 | 中 |
| **项目经理** | 1. 基于模板快速创建新项目计划<br>2. 动态调整项目计划（拖拽、修改依赖）<br>3. 监控关键路径和项目整体进度<br>4. 将计划分派给相关人员 | 高 |
| **部门主管/岗位人员** | 1. 查看自己负责的任务列表和甘特图<br>2. 更新所负责任务的实际开始/完成时间<br>3. 接收任务变更通知 | 中-高 |

### 2.2 业务流程
**正常流程：项目计划全生命周期管理**
1.  **计划编制阶段**：
    - **步骤1（模板准备）**：计划管理员根据历史优秀项目经验，在系统中创建或优化一个计划模板，定义标准任务节点、工期、依赖关系、关联岗位及KCP标识。
    - **步骤2（创建计划）**：项目经理新建项目，选择适用的模板，输入项目名称、计划开始时间。
    - **步骤3（资源匹配）**：系统根据模板中的“关联岗位”，自动列出需要分配人员的岗位列表。项目经理为每个岗位指定具体的人员（从人事系统同步而来）。
    - **步骤4（计划生成）**：系统根据模板结构、计划开始时间、任务依赖关系（FS/SS/FF/SF）及任务工期，自动计算所有任务的计划时间，生成完整的项目计划甘特图，并高亮显示关键路径和KCP节点。
2.  **计划执行与监控阶段**：
    - **步骤5（计划发布与协同）**：项目经理确认计划后，点击“发布”。系统自动将任务通知发送给对应的岗位人员。相关人员可在自己门户查看任务。
    - **步骤6（进度更新）**：岗位人员在实际开始和完成任务时，在系统中更新任务的实际时间。系统自动在甘特图上用不同颜色条形图展示计划 vs 实际进度。
    - **步骤7（计划调整 - 如有变更）**：当发生需求变更、资源变动或进度延迟时，项目经理在甘特图上直接拖拽调整某个任务的计划时间，或修改任务间的依赖关系。
    - **步骤8（关键路径重算）**：系统在计划调整后，自动重新计算整个项目的关键路径，并更新甘特图展示。项目经理可评估调整影响。
    - **步骤9（通知与同步）**：计划调整后，系统自动通知受影响的任务负责人，确保信息同步。

**异常流程：**
- **异常1：人员匹配缺失**：在步骤3，若某个岗位在当前时间点无可用人员（如离职、调岗未同步），项目经理可暂时置空或标记为“待定”，系统应给出明确警告，但不阻止计划生成。
- **异常2：循环依赖**：在步骤1或步骤7，用户设置了A->B->C->A的循环依赖关系。系统在保存时应进行逻辑校验，阻止循环依赖，并给出明确错误提示：“检测到循环依赖，请检查任务[X]与任务[Y]的关系”。
- **异常3：外部系统接口故障**：在步骤3同步人员数据时，若人事系统接口异常，系统应使用本地缓存的最新数据，并记录日志、向管理员告警，提示用户“人员数据可能不是最新”。

### 2.3 使用场景
**场景1：紧急订单项目计划制定（项目经理）**
> 销售部接到一个紧急订单，要求30天内交付。项目经理需要立即制定生产计划。他登录PMS，选择“A类产品标准生产”模板，设定计划开始时间为明天，快速为“设计”、“采购”、“生产组装”等岗位分配好当前空闲的骨干人员。系统在2分钟内生成了详细的甘特图计划，并标出“设计评审完成”和“物料齐套”两个KCP节点。项目经理基于此计划，迅速组织项目启动会。

**场景2：因物料延迟调整生产计划（项目经理）**
> 项目经理收到通知，关键物料“电机”将延迟3天到货。他打开PMS中对应项目的甘特图，找到依赖此物料的“整机装配”任务，将其开始时间向后拖拽3天。系统自动重新计算后续所有依赖任务的时间，并更新关键路径。项目经理发现关键路径因此延长了2天，他立即将情况汇报给管理层，并协调测试团队资源，压缩后续“测试”任务的工期，以尽力追回延迟。

**场景3：查看个人周度工作任务（工艺工程师）**
> 工艺工程师小李每周一早上登录PMS，进入“我的任务”视图。系统以列表和甘特图形式展示了他本周需要完成的“工艺图纸绘制”和“工装夹具验收”两个任务，并清晰显示了任务的计划时间、前置任务（“设计输出完成”）状态。小李看到前置任务已完成，于是开始进行“工艺图纸绘制”工作，并在系统中点击“开始”按钮，记录实际开始时间。

## 3. 功能需求

### 3.1 功能全景
运营计划管理系统（PMS）整体功能架构如下：
```
运营计划管理系统 (PMS)
├── 1. 基础配置中心
│   ├── 1.1 用户与权限管理
│   ├── 1.2 岗位管理
│   └── 1.3 系统设置（如：数据同步配置）
├── 2. 计划模板管理
│   ├── 2.1 模板库视图
│   ├── 2.2 模板编辑器（核心）
│   └── 2.3 模板版本管理
├── 3. 项目计划管理（核心）
│   ├── 3.1 项目列表与筛选
│   ├── 3.2 计划生成器（基于模板）
│   ├── 3.3 计划甘特图视图（核心）
│   ├── 3.4 计划调整与关键路径计算
│   ├── 3.5 计划发布与任务分派
│   └── 3.6 进度更新（实际时间填报）
└── 4. 个人工作台
    ├── 4.1 我的任务列表/看板
    ├── 4.2 我的任务甘特图
    └── 4.3 消息通知中心
```

### 3.2 详细功能列表

#### 功能模块1：基础配置中心
| 功能名称 | 功能说明 | 操作流程（主要） | 输入/输出/规则 |
| :--- | :--- | :--- | :--- |
| **1.1 用户与权限管理** | 管理系统用户账号、角色及功能权限。**假设：** 用户基础信息（姓名、工号、部门）通过人事系统接口同步，本地仅维护其系统角色和状态（启用/禁用）。 | 1. 管理员从“同步用户”按钮，手动或定时从人事系统拉取最新用户列表。<br>2. 管理员为用户分配系统角色（如：项目经理、计划管理员）。<br>3. 管理员可禁用/启用用户。 | **输入：** 来自人事系统的用户数据、本地分配的角色。<br>**输出：** 用户列表。<br>**规则：** 角色与权限组绑定，权限组在后台预先配置。 |
| **1.2 岗位管理** | 维护项目计划中涉及的岗位数据。岗位与人事系统中的职位可能不完全一致，允许自定义。 | 1. 管理员/计划管理员新增、编辑、删除岗位。<br>2. 设置岗位名称、编码、描述。 | **输入：** 岗位名称（必填）、编码（必填，唯一）、描述。<br>**输出：** 岗位列表。<br>**规则：** 已被计划模板引用的岗位不可删除，可标记为“停用”。 |
| **1.3 数据同步配置** | 配置与外部系统（人事、组织架构）的接口参数和同步策略。 | 1. 管理员配置API地址、认证密钥、同步频率（如每日凌晨1点）。<br>2. 可手动触发一次同步。 | **输入：** API配置信息。<br>**规则：** 同步失败需记录详细日志并告警。 |

| :--- | :--- | :--- | :--- |
| **2.1 模板库视图** | 展示所有计划模板，支持按部门、类型筛选。 | 1. 用户进入模板库页面。<br>2. 查看模板列表（名称、所属部门、版本、创建人）。<br>3. 可进行“使用”、“查看”、“复制”、“编辑”操作。 | **输出：** 模板列表。 |
| **2.2 模板编辑器** | **核心功能**。以甘特图或列表形式编辑模板，定义任务节点、依赖、关联岗位。 | 1. 创建新模板或编辑现有模板。<br>2. 添加/删除任务节点，设置：<br>   - 任务名称（必填）<br>   - 基准工期（必填，单位：工作日）<br>   - 是否为KCP节点（复选框）<br>   - 关联岗位（下拉选择，可多选）<br>3. 在甘特图上通过连线或在下拉框中选择，为任务设置依赖关系（FS/SS/FF/SF）。<br>4. 保存模板。 | **输入：** 任务属性、依赖关系。<br>**规则：** <br>1. 必须进行“循环依赖”校验。<br>2. 模板中不设定具体日期，日期在生成具体计划时计算。<br>3. 提供“标准模板”和“部门模板”的可见性权限控制。 |
| **2.3 模板版本管理** | 保存模板的修改历史，支持回滚到旧版本。 | 1. 每次保存模板时，系统自动创建新版本（或提示用户输入版本备注）。<br>2. 在模板详情页可查看历史版本列表。<br>3. 可将某个历史版本设为当前有效版本。 | **输出：** 版本列表（版本号、创建时间、创建人、备注）。 |

#### 功能模块3：项目计划管理（核心）
| 功能名称 | 功能说明 | 操作流程（主要） | 输入/输出/规则 |
| :--- | :--- | :--- | :--- |
| **3.1 项目列表** | 展示用户有权限查看的所有项目，支持高级筛选和搜索。 | 1. 用户进入项目主页。<br>2. 查看项目卡片或列表（项目名、状态、计划起止时间、项目经理）。<br>3. 点击项目进入详情。 | **输出：** 项目列表。支持按项目状态、时间范围、项目经理筛选。 |
| **3.2 计划生成器** | 基于模板快速初始化一个项目计划。 | 1. 点击“新建项目”，输入项目基本信息（名称、描述）。<br>2. 选择一个模板。<br>3. 设置该项目的“计划开始日期”。<br>4. **资源匹配**：系统列出模板中所有关联的岗位，为每个岗位选择具体人员（从同步的用户中过滤）。<br>5. 点击“生成计划”，系统自动计算时间，跳转到甘特图视图。 | **输入：** 项目信息、模板、开始日期、岗位-人员映射。<br>**规则：** 人员选择时，可筛选同一岗位下的用户。 |
| **3.3 计划甘特图视图** | **核心页面**。可视化展示项目计划，支持关键路径高亮、KCP节点特殊标记、计划与实际对比。 | 1. 进入项目后，默认显示甘特图。<br>2. 左侧为任务树/列表，显示任务名称、工期、负责人、状态等。<br>3. 右侧为时间轴条形图，不同颜色表示：<br>   - 计划时间（蓝色）<br>   - 实际时间（绿色）<br>   - 关键路径任务（红色边框）<br>   - KCP节点（菱形图标）<br>4. 支持缩放时间轴（天/周/月）。 | **输出：** 交互式甘特图。<br>**规则：** 关键路径算法需支持四种依赖关系（FS, SS, FF, SF）。 |
| **3.4 计划调整** | 支持在甘特图上直接拖拽调整任务时间，或编辑任务属性。 | 1. **拖拽调整**：鼠标拖动任务条形图，改变其开始或结束时间。系统自动保持其工期不变，并询问是否要同步调整其后续关联任务（智能推导）。<br>2. **编辑任务**：双击任务，可修改其工期、依赖关系、负责人等。<br>3. **保存调整**：任何调整后，系统自动在后台重新计算关键路径，并立即刷新甘特图。 | **输入：** 任务的新时间或新属性。<br>**规则：** <br>1. 拖拽受任务依赖关系约束，不能违反逻辑。<br>2. 修改依赖关系后，必须触发关键路径重算。<br>3. 提供“撤销/重做”功能。 |
| **3.5 计划发布与导出** | 将确认的计划发布给团队成员，并支持导出为Excel。 | 1. 项目经理点击“发布计划”。<br>2. 系统向所有任务负责人发送通知（站内信或集成邮件/钉钉）。<br>3. 在项目任意视图，点击“导出”，选择“导出为Excel”，系统生成包含任务列表、时间、依赖、负责人的标准Excel文件。 | **输出：** 通知、Excel文件。<br>**规则：** 导出的Excel应尽量保持甘特图的视觉结构。 |
| **3.6 进度更新** | 任务负责人更新任务的实际开始/完成时间。 | 1. 负责人在“我的任务”或项目甘特图中，找到自己的任务。<br>2. 点击“开始”或“完成”按钮，系统自动记录当前时间为实际时间。<br>3. 也可手动填写实际开始/完成日期。 | **输入：** 实际开始/完成日期。<br>**规则：** 实际完成日期不能早于实际开始日期。 |

### 3.3 页面说明（核心：计划甘特图视图）
**页面名称：** 项目计划甘特图
**页面路径：** `/project/{id}/gantt`
**主要元素与交互：**
1.  **顶部操作栏**：
    - 按钮：`发布计划`、`导出Excel`、`切换视图（甘特图/列表）`、`设置（显示字段筛选）`。
    - 项目状态指示器。
2.  **左侧任务面板**：
    - 可折叠的任务树状结构，展示WBS。
    - 列包括：`任务名称`、`工期`、`计划开始`、`计划完成`、`实际开始`、`实际完成`、`负责人`、`状态`、`KCP`。
    - 支持对列进行排序、筛选。
    - 支持在此面板直接编辑任务名称、工期、负责人（双击或点击编辑图标）。
3.  **右侧甘特图画布**：
    - 时间轴刻度（可根据顶部缩放控件切换天/周/月）。
    - **计划条形**：浅蓝色条形表示任务的计划时间段。鼠标悬停显示详情。
    - **实际条形**：绿色条形叠加在计划条形上，表示实际进度。长度由实际开始/完成时间决定。
    - **关键路径**：属于关键路径的任务，其计划条形拥有红色粗边框。
    - **KCP节点**：在任务条形的计划完成时间点，用一个红色菱形图标标记。
    - **依赖关系线**：任务间的箭头连线，表示依赖关系。鼠标悬停可显示依赖类型（FS等）。
    - **交互**：
        - **拖拽调整**：拖动任务计划条形的左/右边缘，可调整任务开始或结束时间（工期变化）。拖动条形中间，可平移整个任务（工期不变）。拖拽时，界面有辅助线和提示，显示日期变化量。
        - **创建/修改依赖**：从一个任务的条形拖向另一个任务，可快速创建FS依赖。点击依赖线，可弹出小窗修改依赖类型或删除。
        - **右键菜单**：在条形或任务行上右键，可弹出菜单（添加子任务、前置任务、后置任务、删除、标记为KCP等）。

## 4. 非功能性需求

### 4.1 性能需求
- **页面响应时间**：常规操作页面（列表、表单）加载时间 ≤ 2秒。复杂甘特图视图首次加载时间 ≤ 5秒（对于任务数<500的项目）。
- **操作响应时间**：前端交互（如拖拽、点击）响应 ≤ 200毫秒。后台计算（如关键路径重算）应在 **3秒内** 完成并返回结果。
- **系统并发量**：支持至少 **200** 个用户同时在线操作，50个用户并发进行核心业务操作（如生成计划、调整甘特图）。
- **数据处理时效**：与外部系统数据同步，每日增量同步应在 **10分钟** 内完成。

### 4.2 兼容性需求
- **浏览器**：必须支持 **Chrome 90+** 版本。建议兼容 **Microsoft Edge 90+**、**Firefox 88+**。
- **设备**：支持主流分辨率的桌面端显示器（1920*1080及以上为佳），暂不要求移动端适配。
- **客户端**：纯Web应用，无需安装额外客户端插件。

### 4.3 安全性需求
- **身份认证**：与公司统一身份认证系统（如LDAP/AD）集成，实现单点登录（SSO）。
- **权限控制**：前端路由与后端API均需进行严格的角色与权限校验，防止越权操作。
- **数据安全**：用户密码等敏感信息加密存储。传输过程使用HTTPS。
- **日志审计**：关键操作（如创建项目、调整计划、发布计划）需记录操作日志（Who, When, What, Where），日志保留至少180天。
- **防攻击**：具备基础的防SQL注入、XSS攻击、CSRF攻击的能力。

### 4.4 可用性需求
- **系统可用性**：核心业务功能（计划查看、编辑）月度可用性 ≥ 99.5%。
- **容错能力**：用户操作错误时（如输入非法日期），应有明确、友好的提示信息。网络中断时，前端应有提示，并尝试自动保存未提交的数据（如使用本地存储暂存）。
- **异常提示**：所有系统级错误（如接口超时、服务器错误）应有统一、人性化的错误页面或提示框，避免直接暴露技术栈信息。
- **帮助与引导**：针对“模板编辑”、“甘特图调整”等复杂功能，提供静态图文帮助文档，并考虑在首次使用时增加**关键操作引导**。

## 5. 其他说明

### 5.1 上线计划
根据期望上线时间 **2026-04-27** 进行倒推，建议采用分阶段迭代上线的策略。
- **Phase 1：MVP核心功能上线（2025.12.31）**
    - 里程碑：完成基础配置、模板管理、计划生成与静态甘特图查看、导出Excel功能。
    - 范围：满足“快速生成计划”和“可视化查看”的核心目标，关键路径为静态计算。
- **Phase 2：动态调整与协同（2026.03.31）**
    - 里程碑：完成甘特图动态拖拽调整、关键路径自动重算、任务发布与进度更新功能。
    - 范围：满足“动态调整”和“基础协同”目标，进行UAT用户验收测试。
- **Phase 3：优化与正式上线（2026.04.27）**
    - 里程碑：性能调优、安全加固、用户培训、数据迁移（如有历史项目）、系统正式上线。
    - 范围：系统稳定运行，核心用户完成培训，开始推广使用。

### 5.2 运营建议
1.  **种子用户培养**：在Phase 1开发期间，即邀请2-3位资深项目经理作为种子用户，参与原型评审和MVP版本试用，收集反馈，培养意见领袖。
2.  **分层培训**：
    - **管理员培训**：针对系统管理员和计划管理员，培训系统配置和模板管理。
    - **关键用户培训**：针对所有项目经理，进行集中式、手把手的实操培训，重点讲解模板使用和计划调整。
    - **全员宣导**：通过公司内网、邮件等方式，向所有可能涉及的岗位人员介绍系统价值和使用方法（查看任务、更新进度）。
3.  **激励与推广**：上线初期，可设立“最佳计划模板”奖，鼓励分享模板；将项目计划使用PMS的情况纳入项目经理的绩效考核参考维度之一。
4.  **持续支持**：建立产品反馈渠道（如企业微信群），产品经理和运维团队及时响应问题，并定期（每季度）收集需求，规划后续迭代。
